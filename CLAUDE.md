## CI

CI runs on ubuntu, macOS, and Windows. It enforces: `cargo fmt --check`, `cargo clippy -- -D warnings`, `cargo build --release`, `cargo build --no-default-features`, and `cargo test`. All four must pass.

## Architecture

### Two scan modes

**Targeted scan** (`heft scan`, `heft clean`, bare `heft`): Runs registered `Detector` implementations sequentially. Each detector returns `Vec<BloatEntry>` with category, location, size, and timestamp. After detection, activity checking and staleness scoring annotate every entry.

**Full-drive audit** (`heft audit`): Uses `jwalk` for parallel filesystem walking. Classifies every file via `categories::classify_path()` using a priority chain: custom rules > dev artifact detection > path-based classification > parent context > extension matching. Results display in a ratatui TUI or export as JSON/CSV.

### Default flow (bare `heft`)

`default.rs` orchestrates: runs detectors, finds big files via `big.rs` (jwalk with mount-boundary filtering), deduplicates against detector results, scores everything by staleness, then presents an interactive prompt (clean all stale / pick items / quit).

### Key abstractions

**Detector trait** (`scan/detector.rs`): All scanners implement `detect()` â†’ `DetectorResult`. Four detectors: `ProjectDetector` (node_modules, target/, venv, etc.), `CacheDetector` (npm, cargo, pip, etc.), `DockerDetector` (docker system df), `XcodeDetector` (DerivedData).

**BloatEntry / BloatCategory**: Core data type for scan results. Categories: ProjectArtifacts, PackageCache, ContainerData, IdeData, SystemCache, LargeFile, Other.

**AuditCategory** (`audit/categories.rs`): Separate enum for audit classification (DevArtifacts, Media, Documents, ApplicationData, Logs, etc.). Has a 5-level priority classification chain with `classify_path()`.

**Config** (`config.rs`): Three-layer merge: CLI flags > config file (`~/.config/heft/config.toml`) > defaults. Negative flags (`--no-json`, `--no-verbose`) override config file `true` values. `#[serde(deny_unknown_fields)]` on all config structs.

**Activity detection** (`activity/`): Three signals checked per entry â€” git recency (`git log --since`), source file mtime sampling (breadth-first walk), and running process detection. Returns active/inactive with reason string.

**Staleness scoring** (`staleness.rs`): `size_bytes * factor` where factor comes from configurable age brackets (7dâ†’0.0, 30dâ†’0.5, 90dâ†’1.0, 180dâ†’2.0). Active projects score 0.

### Safety model

All deletion paths must be absolute, under user home or system temp. Symlinks not followed. Docker cleanup uses `docker system prune`, not filesystem ops. Active projects are protected by default (override with `--include-active`).

### Feature gating

TUI (`ratatui`/`crossterm`) is behind the `tui` feature, enabled by default. `cargo build --no-default-features` must compile.

## Identity

Never commit, author, or sign anything as Claude, Anthropic, AI, or any bot identity. All commits use the repository owner's git config. No "generated by", "co-authored-by: claude", or AI attribution anywhere in code, commits, PRs, changelogs, or docs.

## Code Standards

### Style

Every function does one thing. Comments explain why, never what. Use first person singular ("i check" not "we check"). Variable names descriptive enough to read as prose. Error messages help the person who sees them.

Comments: plain english, lowercase, no emojis, no dashes as decorators, no ASCII art, no section banners. Read like margin notes from a coworker.

### Structure

Modules stay under 300 lines. If a file exceeds that, split it. No abstractions until second use. No dependencies without justification in Cargo.toml comments.

### Testing

Test names describe the case: `test_docker_detector_handles_daemon_not_running`. No external services, filesystem state, or network access. Mock at the boundary. Test edge cases and error paths, not obvious behavior.

### What not to do

Do not write code that looks generated â€” no repetitive boilerplate, no overly uniform comment patterns, no alphabetically sorted imports. Do not leave TODO comments referencing conversations or prompts. Do not over-document with contribution guides, codes of conduct, or security policies.

## Git Workflow

`dev` is the integration branch. `main` is stable. Feature branches: `<issue-number>-<description>`. Phase branches (e.g., `p6/snapshot-features`) bundle related features before merging to `dev`.

**Always `--base dev`** when creating PRs. Never PR directly to main from feature branches.

Commit messages: conventional format, lowercase, present tense, under 72 characters. Body explains why, wraps at 72 chars. No emojis. Reference issues with `#N` or `closes #N`.

## Config File

`~/.config/heft/config.toml` with sections: `[scan]`, `[detectors]`, `[activity]`, `[audit]`. CLI flags always override file config.
